# 断点续传协议

## checkpoint.md 字段说明

| 字段 | 说明 |
|------|------|
| 当前步骤 | 当前步骤标识（step-0-init ~ step-done） |
| 状态 | not-started / in-progress / completed |
| 最后更新 | 最后更新时间 |
| 进度明细 | 当前进度描述，1-2 句话 |
| 下一步动作 | 恢复后第一个操作，精确到文件 |
| 已完成步骤 | 步骤 + 完成时间 + 一句话小结 |
| 关键决策 | 时间 + 决策 + 理由 |

## 更新时机

| 事件 | 更新内容 |
|------|---------|
| 开始新步骤 | 当前步骤、状态 → in-progress |
| 完成步骤内关键子任务 | 进度明细 |
| 完成整个步骤 | 状态 → completed、追加「已完成步骤」 |
| 遇到阻断或重大决策 | 进度明细记录原因、追加「关键决策」 |
| 会话结束前 | 必须写入下一步动作 |

## 步骤标识对照表

| 步骤标识 | 阶段名称 |
|---------|---------|
| step-0-init | 初始化 |
| step-1-collection | 检索与留存 |
| step-2-evaluation | 可用性评估 |
| step-3-reasoning | 推理与验证 |
| step-4-writing | 章节化写作 |
| step-5-review | 终稿质检 |
| step-done | 全部完成 |

## 恢复流程

当用户要求继续之前的任务、或检测到项目目录已存在时：

1. **读取 checkpoint.md**：获取当前步骤、状态、下一步动作。
2. **校验关键文件**：检查当前步骤涉及的产出文件是否存在且非空。
   - 文件缺失或损坏：回退到对应步骤重新执行。
   - 文件完整：继续下一步。
3. **按需读取上下文**：
   - `task-brief.md`：回顾任务定义
   - 当前步骤涉及的工作文件
4. **向用户确认恢复**：汇报当前进度和下一步动作，确认后继续。
5. **从 next_action 继续执行**，而非从头开始。

## 步骤回退规则

当后续步骤发现前置步骤的产出不足时，允许回退：

| 场景 | 回退动作 |
|------|----------|
| Step 3 发现来源不足 | 回退到 Step 1，补充检索后重新进入 Step 3 |
| Step 3 发现评估遗漏 | 回退到 Step 2，补充评估后继续 |
| Step 4 写作中发现论证断裂 | 回退到 Step 3，补充推理后继续 |
| Step 5 质检发现事实不可验 | 回退到 Step 1 或 Step 3 |

回退时的 checkpoint 更新：
1. 在「关键决策」表记录回退原因。
2. `当前步骤` 改为回退目标步骤，`状态` = `in-progress`。
3. `进度明细` 标注回退来源（如「从 step-3 回退到 step-1，原因：chapter-02 核心主张缺少来源支撑」）。
4. 补充完成后，重新推进到原步骤继续执行。

## 一致性保障

- **文件存在性优先于状态记录**：如果 checkpoint 标记某步骤已完成但产出文件缺失，视为未完成，回退重做。
