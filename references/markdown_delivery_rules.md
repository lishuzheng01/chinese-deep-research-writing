# Markdown 交付规则

本文件适用于所有正式中文长文交付场景（论文、报告、特稿、专题文等）。

## 1. 默认交付格式

- 最终成果必须以 `.md` 文件形式交付，不得仅以聊天消息作为唯一产物。
- 聊天消息中可附简要说明，但完整内容必须落入本地 markdown 文件。
- 除非用户明确要求 `.docx`、`.pdf` 等格式，否则一律使用 markdown。

## 2. 交付产物与项目结构的关系

交付文件位于标准项目结构的以下路径：

| 类型 | 路径 |
|------|------|
| 章节草稿 | `06-drafts/01-引言.md`、`06-drafts/02-主体.md` 等 |
| 全文汇总稿 | `06-drafts/99-全文汇总.md` |
| 最终交付稿 | `07-delivery/final-manuscript.md` |
| 参考文献 | `07-delivery/references.md` |

交付文件的生成不替代过程文件，二者必须同时存在。

## 3. 检索留痕产物（强制）

每章的 `01-collection/chapter-NN/` 下必须包含：
- `sources/pass-01/`（首轮检索捕获）
- `sources/pass-02/`（补充检索捕获，按需创建）
- `source-index.md`（登记 URL、检索时间、检索轮次、状态 `used` / `unused`、可用性评估与使用方式建议）

来源捕获文件要求：
- 每条来源独立保存为 `Sxxx-来源短名.md`
- 原始文本完整保留，不做改写或精简
- 无论最终是否引用，均需保存并在索引中登记

## 4. 拆分触发条件

拆分阈值为弹性规则，按以下优先级确定：

1. **用户指定**：用户在 `task-brief.md` 中自定义的阈值优先
2. **体裁默认**：
   | 体裁 | 全文阈值 | 单章阈值 |
   |------|---------|---------|
   | 学术论文 | 12000 字 | 3000 字 |
   | 研究报告 | 15000 字 | 3500 字 |
   | 深度特稿/评论 | 8000 字 | 2500 字 |
   | 新闻/解释性报道 | 6000 字 | 2000 字 |
   | 政策分析 | 15000 字 | 3000 字 |
3. **全局默认**：全文 15000 字 / 单章 3000 字

以正文主体长度为估算依据，忽略 markdown 语法开销。

## 5. 拆分方式

触发拆分后：
- 每个一级章节或主要逻辑段落各生成一个 `.md` 文件
- 保持标题与章节顺序不变
- 标题层级在分章文件与汇总文件间保持一致
- 避免过度碎片化（不必将一章再拆成多个小文件，除非用户明确要求）
- 同时生成一个完整的汇总 markdown 文件

## 6. 文件命名规范

使用数字前缀确保排序正确。

示例：
- `00-摘要.md`
- `01-引言.md`
- `02-研究背景.md`
- `03-主体分析.md`
- `04-结论.md`
- `99-全文汇总.md`

非学术写作时，章节名称跟随实际结构，但仍保留数字前缀。

## 7. 汇总文件要求

汇总 markdown 文件必须：
- 包含完整标题
- 章节按正确顺序排列
- 引用编号与参考文献格式保持一致
- 可直接作为单一终稿阅读
- 在章节衔接处修正跨文件过渡问题（如转承语）

## 8. 跨文件引用处理

- 全篇使用统一引用编号体系。
- 除非用户明确要求按章独立编号，否则不得在各章中重新编号。
- 若仅在汇总文件中放置完整参考文献列表，分章文件内的正文引用标记仍须与汇总文件一致。

## 9. 过程文件不可省略

无论交付稿是否已完成，以下过程文件必须在项目目录中保留：
- `00-project-meta/` 下的任务定义
- `01-collection/` 下的检索来源与评估
- `02-reasoning/`、`03-verification/`、`04-academic-validation/` 下的中间分析
- `05-writing-guide/` 下的写作指导
- `90-process-logs/` 下的行为日志

交付完成后不得删除上述过程文件。
